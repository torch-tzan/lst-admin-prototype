# 客戶會議 Demo 腳本（v0.1）

> 日期：2026-04-23（週四）上午　　　　目的：跨 app 管理後台初版對齊
> 原則：**每頁都停下來問客戶「有沒有缺的功能、有沒有多的功能」**，這是定稿級原型要對齊需求的關鍵。

## 場前準備

1. 打開 [http://localhost:3000](http://localhost:3000) → 點「重置 demo 資料」確保乾淨狀態
2. 先登入「LST 平台管理員」走一遍 Dashboard，確認 KPI 數字與爭議賽事都顯示正常
3. 準備 iPad / 第二螢幕：user app / coach app 開在旁邊對照

## 會議開場（3 分鐘）

> 「我們根據兩顆 app 的欄位與流程，整理出管理後台應該長什麼樣子。
> 這個版本是**可操作的互動原型**，今天一起確認：(1) 功能範圍對不對，(2) 哪些地方要再加、哪些可以砍，(3) 兩種角色的權限切分合不合理。」

重點提示客戶兩種角色的差別：**側邊欄會依身份長不一樣、資料範圍也不同**。

## 第一條主線：場館日常營運（場館 admin 視角）⭐

**登入：「パデルコート広島 管理員」**

### 1. Dashboard（1 分鐘）
- 點出本場館「今日 5 筆預約」、「本月營收 ¥XXX」、「3 筆待處理」
- 問客戶：「場館老闆每天早上最想看到什麼？這些 KPI 有缺嗎？」

### 2. 預約管理 → 改期審批（2 分鐘）⭐核心
- 切到 `/bookings`
- 篩選「改期待審」→ 點「佐藤 花子」那筆
- 展示抽屜裡的「改期申請」黃色框：原時段、申請時段、用戶說明
- 按「核准改期」→ 出現 toast、狀態變已確認
- 再選另一筆按「駁回」→ 展示回到原時段的邏輯

### 3. 預約管理 → 退款（2 分鐘）⭐核心
- 切到「退款待審」tab
- 點「渡辺 正」那筆（雨天取消）
- 按「進行退款」→ 表單驗證（金額、理由）→ 確認退款
- Toast 出來、狀態變已取消

### 4. 場地時段 grid（2 分鐘）
- 切到 `/courts`
- 選「コートA」展示星期 × 小時格子
- 點任一格子示範開關
- 按「批次設定」→ 選平日 9-17 → 示範批次更新

### 5. 設備租賃（1 分鐘）
- 切到 `/equipment`
- 指出「タオル剩 2 件」的低庫存黃色警示
- 按「新增設備」→ 帶客戶走一次表單（特別強調「按時 vs 按次計價」選項）

### 6. 場館教練 + 評價（1.5 分鐘）
- 切到 `/venue-coaches`
- 切到「評價管理」→「待回覆」
- 點開 1 星差評 → 按「回覆」→ 填 → 送出
- 點「申請下架」展示下架 confirm dialog

## 第二條主線：LST 平台跨場館治理（平台 admin 視角）⭐

**登出 → 登入「LST 平台管理員」**

提醒客戶：**側邊欄變了**—多出「場館管理、教練審核、賽事審核、公告推播」。

### 1. Dashboard（1 分鐘）
- 指出跨場館 KPI：今日預約數合計、活躍教練、爭議賽事、營運中場館
- 點「待處理：教練審核」區塊直接跳頁

### 2. 教練審核（2 分鐘）⭐核心
- 四個 tab：待審核 / 已通過 / 已退回 / 停權
- 點「中村 健」待審教練 → 抽屜展示自我介紹、專長、證照、匯款帳戶
- 按「通過審核」→ confirm dialog → 完成，教練進「已通過」tab
- 回列表點另一位 → 按「退回」→ 表單要求至少 10 字理由 → 送出

### 3. 賽事審核（1.5 分鐘）
- 切到 `/matches`「爭議」tab
- 點「Hiroshima Smashers vs Kita Rising Stars」
- 抽屜展示比分、爭議理由
- 按「送出裁決」→ 選勝方、填比分、寫裁決說明 → 確認

### 4. 場館管理（1 分鐘）
- 切到 `/venues`
- 按「新增場館」快速示範表單（名稱、地區、聯絡人、email 驗證）
- 對既有場館按「停用」→ 狀態即變

### 5. 公告推播（2 分鐘）⭐核心
- 切到 `/announcements`
- 按「新增公告」
- 示範：
  - 標題少於 4 字會擋下（驗證）
  - 下方「推播預覽」會即時更新
  - 選「排程送出」→ 會多出時間欄位
  - 送出後 toast

## 第三條主線：資料範圍切分驗證

**登出 → 登入「北広島パデルクラブ 管理員」**

- 直接切到 `/bookings`：**看不到 パデルコート広島 的預約**
- 切 `/courts`：**只看得到自己場館的 2 座場地**
- 嘗試手動在網址輸入 `/announcements`：**側邊欄沒有、會被 layout 擋回**（此處用來強調權限是後端也要做的，前端只是提示）

## 收尾對話（5 分鐘）

問客戶：
1. **功能範圍**：有什麼是目前後台**沒想到但一定要有**的？（例：財務報表、折價券、客戶黑名單、電子發票…）
2. **權限細節**：場館 admin 能看到營收嗎？能看到教練的銀行帳戶嗎？低分評價下架是否要 LST 審核？
3. **資料欄位**：哪個表格缺了欄位？哪些目前存在的欄位其實用不到？
4. **v2 優先級**：若下一輪可以挑 2 個模組做高保真 + 真後端，客戶最想看到哪兩個？

## 備用：如果客戶問「這個怎麼做到的」

- 純前端 Next.js 14 + Tailwind，資料用 localStorage 模擬
- 登入是點擊測試帳號（沒有真 auth）
- 下一輪要接 Supabase：Auth + RLS + Edge Functions，跟兩顆 app 共用同一個 DB
- 預計開發時程（v1 正式版）：設計 2 週 + 前端 3 週 + 後端整合 2 週 ≈ 7 週可上線
